# Architecture

The application separates reusable automation behavior from the WPF shell:

```text
Core <- Vision ----\
  ^                 \
  +---- Windows -----> Automation -> App
  +-------------------------------> App
```

Project references are allowlisted and checked by `scripts/Test-RepositoryPolicy.ps1`. Adding a project or dependency requires an explicit architecture-policy update; lower layers must never acquire a reference to a higher layer.

- `ExpeditionsMacro.Core` contains immutable geometry, image, model, preset, persistence, and workflow contracts.
- `ExpeditionsMacro.Windows` owns Win32 window discovery, exact client sizing, Windows Graphics Capture of the Roblox compositor surface, FP16 scRGB-to-SDR conversion, bounded GDI capture for the explicit desktop region selector, DPAPI, configurable key input, Roblox-compatible mouse nudging and acknowledged client-control drags, Roblox `O`-key zoom with a wheel fallback, extended scan-code camera arrows, and relative right-drag fine camera input. The global hotkey accepts safe single-key letters, digits, punctuation, numpad keys, and supported function keys. Shift Lock resolves the saved virtual key to a physical scan code and preserves extended-key identity, so Left/Right Shift and Left/Right Ctrl remain distinct.
- `ExpeditionsMacro.Vision` owns lighting-normalized scoring, temporal medians, detector-pack compilation, validation, classification, Expedition and Challenge states, Story/Raid stage navigation, saved-team dialogs, map and difficulty selection, node hues, and hotbar checks.
- `ExpeditionsMacro.Automation` owns camera calibration/alignment, placement recording/playback, saved-team loading, the Expedition, Challenge, Story, and Raid runners, the prioritized Macro scheduler, recovery, Discord Components V2, and detector updates.
- `ExpeditionsMacro.App` is the single WPF interface and exclusive input-owner coordinator.
- `ExpeditionsMacro.DatasetBuilder` compiles a local capture corpus into a small, hashed detector pack.

All saved coordinates are relative to the Roblox client. Camera calibration and placement recording use the canonical 808 by 611 detector size. Camera setup standardizes zoom and pitch, snapshots the globally configured Shift Lock key for balanced enable/cleanup, and selects stable map regions while preserving left/center/right evidence. Schema 4 calibration captures the signed fine-yaw neighborhood with atomic gestures, treats its verified stationary return as the authoritative zero, then holds Right Arrow once and samples a dense full turn at up to roughly 60 frames per second. A compact regional luminance/texture/gradient descriptor locates the returning neighborhood; acceptance additionally requires either a matching signed fine-yaw reference or an exact registered structural match. Because the game can continue rotating between the recognized callback frame and key release, setup then locates the stationary pose in the completed atlas and returns through bounded, re-observed arrow groups before fine verification. A bounded discrete-pulse probe searches only physically possible forward atlas ranges and measures the conversion between dense visual bins and arrow pulses, rather than assuming continuous hold time and repeated taps have identical rates. Its return uses the same atlas feedback instead of assuming equal opposing pulse counts are visually reversible. Normal setup has a tested sub-20-second performance goal, while a separate 120-second internal cancellation boundary handles a permanently stalled capture or game; model saving remains transactional. Schema 3 pulse atlases remain loadable.

Runtime alignment converts a dense atlas match back to pulse distance and uses the saved full-turn atlas for coarse correction plus the saved fine atlas before its live micro-search. Coarse correction is closed-loop: it sends at most six arrow pulses, re-observes the actual atlas position, and recalculates the shortest remaining direction after each group. A dense fingerprint may earn this bounded feedback path when it is both strong and isolated from every remote yaw neighborhood, even if the continuously moving setup frame has lower registered structure; weak structure, a non-unique fingerprint, stationary evidence, or a position cycle stops that loop and retains the existing bounded verified scan. Fingerprint evidence never completes alignment: signed fine-yaw refinement and the three independently rendered full-resolution goal frames remain authoritative. After two normal macro runs observe the same stabilized load-in fingerprint, a device-local shortcut can apply one learned relative mouse drag and verify the goal across three frames; any mismatch falls back to the complete atlas path, repeated failures relearn the shortcut, and manual Auto Align neither trains nor consumes it. Thumbnail atlas lookup retains registered structural reranking so adjacent or visually aliased yaw remains distinct. Full-resolution goal verification may accept a larger vertical projection shift only when at least three of the four saved regions independently agree on the same translation; after any complete arrow-key turn and return, the engine re-observes the live pose before applying its saved fine correction. Playback and alignment apply the model's recorded client size. When the app or a workflow resizes Roblox, it leaves Roblox at that standardized client size so later detection and placement steps share the same geometry. Window discovery verifies the owning Roblox player process and carries its PID into run diagnostics. A stale handle is rediscovered after focus failure and transparently aliased for later capture and input. Normal framed sizing remains the first path; if Windows clamps it above the canonical client size, the Windows layer removes minimum-tracking frame styles, applies a verified borderless size, and retains the original styles for lifecycle cleanup.

The workflow never reads Roblox memory or injects into the process. It observes screen pixels and emits ordinary Windows keyboard and mouse input.

Detector captures target the Roblox HWND through Windows Graphics Capture rather than copying the desktop rectangle at the same coordinates. Occluding applications are therefore excluded. The capture frame pool uses `R16G16B16A16_FLOAT`; linear scRGB pixels are locally tone-mapped and sRGB-encoded before becoming the same 8-bit `ImageFrame` consumed by vision. SDR pixels below the highlight shoulder preserve the standard transfer curve, while HDR/Auto HDR highlights are compressed instead of clipped. Desktop GDI capture remains limited to the user-driven camera-region selection overlay.

Frame-arrival callbacks remain notification-only for Windows 10 compatibility. The serialized capture path drains any queued compositor frames and scores only the newest one, so settled camera input cannot be compared against an older pose. If the post-barrier frame wait expires during a Roblox compositor handoff, the Windows layer disposes that capture session, creates one replacement with the longer initial-frame allowance, and retries without emitting any additional game input. A second timeout remains a session-level failure.

Play navigation uses a separately configured A-Z key matching Anime Expeditions' **Toggle Play Menu** binding. The runner presses it from the lobby, terminal, or prestart state, verifies the expected selector or party preview, and retries boundedly; it does not depend on the position or visibility of either on-screen Play button.

Actionable navigation waits combine state stability with geometry stability. A detector-owned action must keep the same state and remain within three client pixels for at least two consecutive observations before input; a single late Play frame never bypasses that waiter. Unit Team loading additionally rejects neutral-gray candidates outside the field-observed scrollbar-thumb height, waits for the real thumb to stop moving, and normalizes a non-top reopening before aligning the requested team.

Macro plans persist immutable task definitions separately from per-task progress. The scheduler selects the lowest priority number among enabled, incomplete, and currently eligible tasks. A runner owns Roblox input until it returns a result; only then does the scheduler persist progress and choose again. Challenge tasks remain recurring and carry a next-eligible timestamp, while Expedition, Story, and Raid tasks complete against saved victory or Infinite-runtime targets. One operation-scoped reporting accumulator owns total runtime, victories, and defeats across every mode handoff and process-recovery retry; individual result cards retain the runtime for that specific match. The outer recovering scheduler does not mutate task progress for a failed attempt. When enabled, it classifies session-level failures, asks the Windows process adapter to close only the verified Roblox PID, relaunches a normalized private-server URI through the registered Windows protocol, reloads the saved plan, and retries the same incomplete task. A three-restarts-per-ten-minutes circuit breaker bounds this process-level fallback.

Scheduled mode handoffs use shared, verified navigation screens rather than leaving an owner-specific panel for the next runner. Challenge may retain its selector only between immediately consecutive Challenge matches; once the rotation enters cooldown, it closes that selector and verifies the game-mode selector before returning. Scheduled Expeditions returns at a verified Play party preview, which every mode entry can convert to the game-mode selector. Story and Raid return at the verified game-mode selector after both terminal results and recoverable camera-alignment skips. If a handoff cannot be verified after bounded retries, the plan stops instead of starting another mode from an unknown screen.

Long-running gameplay monitors and Challenge cooldown waits schedule an ordinary `O` pulse every eight minutes through the shared focused-keyboard path. Input failures defer the next attempt for one minute without terminating the run. After an AFK-Chamber Return-to-Lobby action, Story/Raid navigation suppresses Play-key attempts throughout the teleport/loading interval and resumes only from a verified lobby or game-mode selector.

Story and Raid navigation uses specialized structural detectors for the mode selector, map selector, detail panel, party preview, prestart, terminal screens, and saved-team dialogs. Mode-detail detection treats **Enter Matchmaking** as optional because fresh lobby parties expose it while post-match parties retain only **Select Stage**; actions always map through the detected live button. A launch-ready party is capability-based: both the lobby **Disband** family and post-match **Change Gamemode** family are accepted only when a live **Start** action is independently detected, while handoff logic continues to distinguish their different exit controls. Team loading is a separate service that validates the canonical client size before every capture and action. It detects and holds the neutral gray Unit Teams scrollbar thumb, drags the requested row into a field-confirmed aligned position, and verifies a fully visible Load button before clicking; relative wheel input remains available for other bounded list navigation.

Deep debug logging decorates the shared `IRobloxAutomation` boundary and the exclusive operation coordinator. The Windows implementation emits exact low-level key/mouse events, the vision layer emits detector scores and selected states, and the app forwards workflow progress and decisions. A bounded writer preserves ordering and applies backpressure rather than dropping frames. Each enabled operation snapshots a secret-free configuration graph and finalizes its own Zip64-capable archive for success, cancellation, or failure; if finalization fails, its staging directory is intentionally retained. One archive-text owner redacts webhook/private-server secrets, Discord user IDs, and the active Windows username/profile path from serialized events, logs, errors, and copied text model/configuration files before they enter that archive.

The optional Debug workspace wraps that same automation boundary with a checkpoint controller. It retains the newest captured frame, records detector and action checkpoints, and may gate either input actions or every detector observation. A gate only delays the owning operation; it never synthesizes an undo. Timeline rewind changes the WPF preview selection, while Step releases exactly the current live gate and Run removes subsequent gates. Debug navigation calls mode-owned navigation entry points and stops at verified prestart, and team testing calls the production saved-team service, so diagnostic tools cannot drift into separate blind-click implementations. Debug operation context, checkpoints, frames, detections, and inputs flow through the existing Deep Debug archive when enabled.

Camera readiness is a separate pre-alignment gate over the saved world regions. It rejects the field-confirmed textureless blue load state before any yaw or unit placement. Mode runners first perform their verified in-client Play/navigation recovery and retry the same route; a resulting bounded timeout may then reach the outer process-restart policy. Fine-yaw movement is decomposed into identical atomic right-drag gestures, so calibration and runtime correction do not mix one multi-pixel gesture with repeated one-pixel gestures on devices where Roblox consumes those paths differently.
