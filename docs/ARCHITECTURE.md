# Architecture

The application separates reusable automation behavior from the WPF shell:

```text
Core <- Vision ----\
  ^                 \
  +---- Windows -----> Automation -> App
  +-------------------------------> App
```

Project references are allowlisted and checked by `scripts/Test-RepositoryPolicy.ps1`. Adding a project or dependency requires an explicit architecture-policy update; lower layers must never acquire a reference to a higher layer.

- `ExpeditionsMacro.Core` contains immutable geometry, image, model, preset, persistence, and workflow contracts.
- `ExpeditionsMacro.Windows` owns Win32 window discovery, exact client sizing, Windows Graphics Capture of the Roblox compositor surface, FP16 scRGB-to-SDR conversion, bounded GDI capture for the explicit desktop region selector, DPAPI, the configurable global macro hotkey, keyboard input, Roblox-compatible mouse nudging, Roblox `O`-key zoom with a wheel fallback, extended scan-code camera arrows, and relative right-drag fine camera input. The global hotkey accepts safe single-key letters, digits, punctuation, numpad keys, and supported function keys.
- `ExpeditionsMacro.Vision` owns lighting-normalized scoring, temporal medians, detector-pack compilation, validation, classification, Expedition and Challenge states, Story/Raid stage navigation, saved-team dialogs, map and difficulty selection, node hues, and hotbar checks.
- `ExpeditionsMacro.Automation` owns camera calibration/alignment, placement recording/playback, saved-team loading, the Expedition, Challenge, Story, and Raid runners, the prioritized Macro scheduler, recovery, Discord Components V2, and detector updates.
- `ExpeditionsMacro.App` is the single WPF interface and exclusive input-owner coordinator.
- `ExpeditionsMacro.DatasetBuilder` compiles a local capture corpus into a small, hashed detector pack.

All saved coordinates are relative to the Roblox client. Camera calibration and placement recording use the canonical 808 by 611 detector size. Camera setup standardizes zoom and pitch, selects stable map regions automatically while preserving left/center/right evidence, learns and persists a signed fine mouse-controlled neighborhood around the goal, then recognizes a full yaw turn with Right-arrow pulses against those nearby views before performing final mouse refinement. Runtime alignment uses the saved full-turn atlas for coarse correction and the saved fine atlas before its live micro-search. After two normal macro runs observe the same stabilized load-in fingerprint, a device-local shortcut can apply one learned relative mouse drag and verify the goal across three frames; any mismatch falls back to the complete atlas path, repeated failures relearn the shortcut, and manual Auto Align neither trains nor consumes it. Thumbnail atlas lookup retains a tight five-pixel registration window so adjacent yaw remains distinct. Full-resolution goal verification may accept a larger vertical projection shift only when at least three of the four saved regions independently agree on the same translation; after any complete arrow-key turn and return, the engine re-observes the live pose before applying its saved fine correction. Playback and alignment apply the model's recorded client size. When the app or a workflow resizes Roblox, it leaves Roblox at that standardized client size so later detection and placement steps share the same geometry. Window discovery verifies the owning Roblox player process and carries its PID into run diagnostics. A stale handle is rediscovered after focus failure and transparently aliased for later capture and input. Normal framed sizing remains the first path; if Windows clamps it above the canonical client size, the Windows layer removes minimum-tracking frame styles, applies a verified borderless size, and retains the original styles for lifecycle cleanup.

The workflow never reads Roblox memory or injects into the process. It observes screen pixels and emits ordinary Windows keyboard and mouse input.

Detector captures target the Roblox HWND through Windows Graphics Capture rather than copying the desktop rectangle at the same coordinates. Occluding applications are therefore excluded. The capture frame pool uses `R16G16B16A16_FLOAT`; linear scRGB pixels are locally tone-mapped and sRGB-encoded before becoming the same 8-bit `ImageFrame` consumed by vision. SDR pixels below the highlight shoulder preserve the standard transfer curve, while HDR/Auto HDR highlights are compressed instead of clipped. Desktop GDI capture remains limited to the user-driven camera-region selection overlay.

Play navigation uses a separately configured A-Z key matching Anime Expeditions' **Toggle Play Menu** binding. The runner presses it from the lobby, terminal, or prestart state, verifies the expected selector or party preview, and retries boundedly; it does not depend on the position or visibility of either on-screen Play button.

Macro plans persist immutable task definitions separately from per-task progress. The scheduler selects the lowest priority number among enabled, incomplete, and currently eligible tasks. A runner owns Roblox input until it returns a result; only then does the scheduler persist progress and choose again. Challenge tasks remain recurring and carry a next-eligible timestamp, while Expedition, Story, and Raid tasks complete against saved victory or Infinite-runtime targets.

Scheduled mode handoffs use shared, verified navigation screens rather than leaving an owner-specific panel for the next runner. Challenge may retain its selector only between immediately consecutive Challenge matches; once the rotation enters cooldown, it closes that selector and verifies the game-mode selector before returning. Scheduled Expeditions returns at a verified Play party preview, which every mode entry can convert to the game-mode selector. Story and Raid return at the verified game-mode selector after both terminal results and recoverable camera-alignment skips. If a handoff cannot be verified after bounded retries, the plan stops instead of starting another mode from an unknown screen.

Story and Raid navigation uses specialized structural detectors for the mode selector, map selector, detail panel, party preview, prestart, terminal screens, and saved-team dialogs. Mode-detail detection treats **Enter Matchmaking** as optional because fresh lobby parties expose it while post-match parties retain only **Select Stage**; actions always map through the detected live button. Team loading is a separate service that validates the canonical client size before every capture and action. The Windows layer supplies bounded relative wheel input for deterministic team and map-list positioning.

Deep debug logging decorates the shared `IRobloxAutomation` boundary and the exclusive operation coordinator. The Windows implementation emits exact low-level key/mouse events, the vision layer emits detector scores and selected states, and the app forwards workflow progress and decisions. A bounded writer preserves ordering and applies backpressure rather than dropping frames. Each enabled operation snapshots a secret-free configuration graph and finalizes its own Zip64-capable archive for success, cancellation, or failure; if finalization fails, its staging directory is intentionally retained.
